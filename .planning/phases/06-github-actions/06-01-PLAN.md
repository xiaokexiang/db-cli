<task type="auto">
  <name>Task 1: 重构 workflow job 架构</name>
  <files>.github/workflows/release.yml</files>
  <action>
重构 release.yml 为正确的 2-job 架构：

**当前问题:**
- `release` job 尝试使用 `${{ github.event.release.upload_url }}` 上传 assets
- 但 release 是由 `create-release` job 创建的（`needs: release`）
- 这导致 upload step 的 upload_url 为空，必然失败

**修复方案：2-job 架构**

1. **build job** (多平台矩阵构建):
   ```yaml
   build:
     runs-on: ${{ matrix.os }}
     steps:
       - 构建二进制文件
       - 上传为 GitHub Actions artifacts: uses actions/upload-artifact@v4
   ```

2. **create-release job** (创建并发布):
   ```yaml
   create-release:
     needs: build
     runs-on: ubuntu-latest
     steps:
       - 下载所有 artifacts: uses actions/download-artifact@v4
       - 创建 Draft Release: uses actions/create-release@v1
       - 上传 binaries 到 Release: uses actions/upload-release-asset@v1
       - 发布 Release (删除 isDraft 或更新)
   ```

**具体修改:**
1. 移除 `release` job 中的 `upload-release-asset` step
2. 移除 `release` job 中的 `upload-artifact` step (改为在构建后上传)
3. 在 `create-release` job 中：先创建 release → 下载 artifacts → 上传到 release
4. 删除 `if: always()` 条件
5. 权限精简为仅 `contents: write`
  </action>

<task type="auto">
  <name>Task 2: 修复 changelog 生成逻辑</name>
  <files>.github/workflows/release.yml</files>
  <action>
修复 changelog 生成步骤，处理首个 tag 的边界情况：

1. **当前问题**: `git describe --tags --abbrev=0 HEAD^` 在首个 tag 时会失败（没有上一个 tag）

2. **修复方案**: 使用更健壮的条件判断
   ```bash
   # 获取上一个 tag，如果不存在则使用仓库第一个 commit
   PREV_TAG=$(git describe --tags --abbrev=0 HEAD^ 2>/dev/null || git rev-list --max-parents=0 HEAD)
   git log ${PREV_TAG}..HEAD --pretty=format:"- %s"
   ```

3. **或者简化方案**: 直接使用最近 N 条 commit
   ```bash
   git log --oneline -10 --pretty=format:"- %s"
   ```
